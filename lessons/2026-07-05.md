# Lessons — 2026-07-05

## Blueprint interface ownership enforcement

### re.sub backslash in replacement string raises PatternError

- **Context**: `sync_skill_blueprints.py` — `sync_interface_block()` calls `pattern.sub(interface_block, text)`
- **Observation**: When `interface_block` contains any `\X` sequence (e.g. `\i` from a usage string like `[--unit-dir UNIT_DIR]`), Python's `re` interprets it as a backreference and raises `re.PatternError: bad escape`.
- **Impact**: The sync script crashes silently on any blueprint whose usage string contains a backslash.
- **Fix**: Always use `lambda _: replacement` for literal string replacements in `re.sub`. Never pass a user-supplied string directly as the replacement argument.

### `usage: ""` collapses to None through `.strip()`

- **Context**: `sync_skill_blueprints.py` — `owner_interfaces()` extracts the `usage` field.
- **Observation**: The original `usage.strip() if isinstance(usage, str) and usage.strip() else None` collapses `""` (explicitly set, no args) to `None` (not set), because `"".strip()` is falsy. Both render as `...` stubs instead of `""` rendering a clean invocation.
- **Impact**: No-arg interfaces (e.g. `scripts-healthcheck`) always show `...` in the generated block even when `usage: ""` is explicitly declared.
- **Fix**: Preserve the distinction explicitly — `None` if key absent, `usage.strip()` (may be `""`) if key present. The rendering logic then branches on `usage is None` vs `usage == ""` vs non-empty string.

### Dependency validator scans generated blueprint blocks for skill names

- **Context**: `dependencies.py` — `_strip_frontmatter_and_deps()` is used before scanning the SKILL.md body for skill-name mentions.
- **Observation**: The function stripped frontmatter and the Dependencies block but not the `<!-- BEGIN/END BLUEPRINT INTERFACES -->` generated block. Interface IDs like `runners-daily-plan` contain skill names (`daily-plan`) as substrings, triggering false "body mentions" that mismatched the sidecar.
- **Impact**: Skills with interface IDs that embed another skill's name (e.g. `runners-daily-plan` in recurring-tasks) fail the dependency validator with a spurious mismatch.
- **Fix**: Add `_BLUEPRINT_BLOCK_RE` stripping in `_strip_frontmatter_and_deps()` to remove both CONTRACT and INTERFACES generated blocks before the mention scan.

### Auto-migration `interface:` / `exported_interface:` keys are schema violations

- **Context**: All blueprints created by the legacy migration tool use `interface:` and `exported_interface:` inside `script_interfaces` entries. The schema has `additionalProperties: false`.
- **Observation**: These keys were silently accepted because `blueprints.py` did not run jsonschema validation — it only checked marker balance and sync drift. The violation was invisible until jsonschema was wired in.
- **Impact**: Blueprints appeared valid but were rejected by any jsonschema-aware consumer.
- **Fix**: Replace `interface: { default: { min_positionals, allow_extra_positionals, ... } }` + `exported_interface: []` with `default: { patterns: [{ name: owner, allow_extra_positionals: true }] }`. Drop `exported_interface` entirely. The `blueprints.py` validator now runs `jsonschema.Draft7Validator` first, catching this and all other structural violations before Python-level checks.

### Internal interfaces: omit description to hide from generated block

- **Context**: `sync_skill_blueprints.py` — `generated_interface_block()` loop.
- **Observation**: All interfaces were previously emitted into the generated block regardless of whether they had a `description`. Internal interfaces (libraries, sourced scripts, systemd entry points) cluttered the model-facing block with entries the model should never invoke directly.
- **Decision**: Interfaces without `description` are now silently skipped by the sync script and not checked by the dispatcher validator. They remain in `blueprint.yaml` for dispatcher enforcement (access control) without appearing in SKILL.md.
- **Use**: To hide an interface from the generated block, simply omit its `description` field. To expose it, add `description` (and the now-required `usage`).

## Cross-host hook installation: dev mode and plugin mode need different host-selection strategies

**Context:** `llmhooks/`, `hooks/hooks.json`, `hooks/inject_dispatcher_context.py`, `skills/install-assistant-tools/scripts/setup_tools.py`

**Observation:** The durable design split is:
- **dev mode** — installer writes explicit host-selected commands such as `python3 .../llmhooks/inject_dispatcher_context.py --claude` and `--codex`
- **plugin mode** — one shared `hooks/hooks.json` serves multiple hosts, so plugin installs still need a thin compatibility shim in `hooks/inject_dispatcher_context.py` that chooses the host adapter from plugin-root variables

**Impact:** Trying to force plugin mode into the same fully explicit command pattern as dev mode creates confusion, because plugin registration currently has one shared hook file but dev mode has host-specific install surfaces. The stable contract is: real hook logic in `llmhooks/`, plugin glue in `hooks/`.

**Best bet:** When adding or debugging a cross-host hook, first ask which path is failing:
- dev install (`settings.local.json` / `config.toml`) → inspect `llmhooks/registry.py` and installer output
- plugin install (`hooks/hooks.json`) → inspect the shim and host plugin-root variables

## Claude plugin SessionStart can be tested before auth by using stream-json hook events

**Context:** `skills/install-assistant-tools/tests/test_claude_install.py`, Claude plugin installs under a temp `CLAUDE_HOME`

**Observation:** A real isolated Claude plugin install can confirm hook firing without a logged-in account by running:

`claude -p hello --output-format stream-json --include-hook-events --verbose --allowedTools ""`

The command still emits `hook_started` and `hook_response` events for `SessionStart` before the final auth failure.

**Impact:** This gives a true end-to-end plugin-mode validation surface for hooks. Without it, plugin-hook confidence is only inferential from installed files and local precedents.

**Use:** When validating Claude plugin hooks in CI-style temp homes, assert:
- `hook_started` exists for `SessionStart`
- `hook_response` contains the expected injected context
- ignore the final unauthenticated failure as long as the hook events were emitted first
