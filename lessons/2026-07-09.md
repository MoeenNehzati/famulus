## Codex GitHub plugin installs preserve skills but not installed-cache symlink aliases

**Context:** `skills/install-assistant-tools/tests/test_codex_github_install.py` while debugging CI failure on missing `installed_path / "AGENTS.md"` after `codex plugin add famulus@nullkit`.

**Observation:** The `AGENTS.md -> CLAUDE.md` symlink survives both `git clone` of the public repo and `codex plugin marketplace add` into the marketplace cache. It disappears only in Codex's final installed plugin cache after `codex plugin add`. The same install also drops other symlinked cache aliases such as `.codex/skills`, `.codex/references`, and `.codex/agents`. Despite that cache-shape change, the installed plugin still exposes Famulus skills correctly in `codex debug prompt-input`.

**Impact:** A test that asserts installed-cache symlink presence can fail even when the published Codex plugin works. The meaningful contract is skill availability and required real assets, not preservation of symlink aliases inside Codex's installed cache.

**Use/Avoid:** When testing Codex GitHub/plugin installs, avoid asserting `installed_path/AGENTS.md` or other installed-cache symlink aliases unless the product explicitly promises that cache shape. Prefer assertions that the plugin installs, required non-symlink assets exist, and skills remain visible/usable.

## Anchored `PurePosixPath.match` is the stdlib replacement for `full_match` on the Python 3.11 CI baseline

**Context:** `skills/cloud-files/scripts/cloud_files.py` glob matching for `ls_llm.py` and `rm_llm.py`, after CI failed with `AttributeError: 'PurePosixPath' object has no attribute 'full_match'`.

**Observation:** Local development was on Python `3.13`, where `PurePosixPath.full_match` exists, but CI runs Python `3.11.9`, where it does not. Replacing it with plain `match("scratch/*")` would be wrong because relative patterns can match from the right. Anchoring both sides with a leading slash — `PurePosixPath(f"/{entry.path}").match(f"/{normalized}")` — preserved the intended whole-path glob behavior using stdlib only.

**Impact:** Local green tests on a newer interpreter can hide portability bugs when code relies on newly added `pathlib` APIs. The safe replacement here was not `fnmatch` or a custom matcher, but anchored `PurePosixPath.match`.

**Use/Avoid:** On this repo's Python baseline, avoid `PurePosixPath.full_match`. When you need whole-path matching with stdlib only, anchor both the candidate path and pattern before calling `PurePosixPath.match`.
