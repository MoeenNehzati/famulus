# Lessons — 2026-07-06

## Non-interactive `bash -lc` never loads conda — jobs get system python3

- Context: `skills/recurring-tasks/scripts/run-skill.sh` executes job commands
  via `exec bash -lc`; there is no `~/.bash_profile`, so `.profile` →
  `.bashrc`, but `.bashrc` early-returns for non-interactive shells *before*
  the conda init block (line ~200).
- Observation: systemd/cron jobs therefore see `/usr/bin/python3` and none of
  `anaconda3/bin` — anything installed only into the conda env (as the
  pip-installed `dispatcher` was) is invisible to jobs while working fine in
  every interactive shell.
- Impact: jobs "succeed" (exit 0) while all dispatcher-backed skills silently
  degrade; nothing fails loudly.
- Use: never rely on conda-env binaries from services. The generated
  `dispatcher` launcher (managed bin dir, runs from `$AI` with system
  `python3`) is the fix; system python needs only PyYAML (apt-provided).

## Repo-mutating test failures surface as job breakage hours later

- Context: `setup_tools.run()` and `uninstall.py` write/delete repo-scoped
  artifacts (`skills/recurring-tasks/scripts/env.sh`, git hooksPath). Tests
  calling them without a throwaway `repo_root` mutate the live checkout.
- Observation: the symptom was never a test failure — it was recurring jobs
  failing at their next scheduled fire (`invoke-agent.sh: env.sh: No such
  file`), hours after the pytest run that caused it. Happened 4× on
  2026-07-05/06 before the cause was pinned.
- Impact: symptom→cause link is non-obvious; the natural debugging entry
  point is recurring-tasks, not the installer test suite.
- Use: if a generated machine-local file under the repo goes missing or gains
  pytest tmp paths, suspect a test run first. Regression test now guards
  env.sh (`test_removes_fake_repo_env_sh_not_real_one`); in-process installer
  calls must pass `repo_root` (see `setup_tools.run()` docstring).

## Other machines still on the pre-2026-07-05 install layout

- Context: this machine got the new layout manually during the work; the
  installer changes (commits b86a46d, 7aa4dc0) are what other machines lack.
- Observation: any other machine still has symlinked profiles (Codex trust
  state leaking into the repo working tree), a pip-installed dispatcher
  (invisible to systemd jobs), and the plugin installed under its old
  pre-rename ID (now `famulus@nullkit`).
- Impact: on those machines, commits will show dirty profiles, recurring jobs
  run degraded, and plugin updates resolve to a dead ID.
- Use: on each other machine, re-run `install-assistant-tools` (converts
  profile symlinks to copies, writes the dispatcher launcher and env.sh) and
  re-install the plugin under the new name.
