# 2026-07-07

## Deriving chars-per-token from Claude Code `usage.output_tokens` is invalid, not just noisy

**Context:** `skills/find-handoff-candidates/_rtx/_handoff_scan.py` — while calibrating the `gap_net_chars` metric, tried to check whether a chars/4 ≈ tokens heuristic was reasonable for this JSON-heavy transcript data (as opposed to plain English prose, where it's a standard approximation).

**Observation:** Attempted to derive an empirical chars-per-token ratio by comparing each assistant message's `usage.output_tokens` against the character length of its own content. Got a ratio under 1 char/token — impossible for real text. Root cause: Claude Code's extended-thinking mode bills reasoning tokens in `usage.output_tokens`, but the actual reasoning content is redacted — only an opaque `signature` field is stored (no plaintext). A large fraction of billed output tokens therefore have zero character representation anywhere in the transcript file to divide against.

**Impact:** Any attempt to calibrate a token-based version of this skill's threshold (or any metric derived from Claude Code transcripts) by comparing `usage.*` token counts against transcript character counts will silently produce a nonsensical ratio, because hidden thinking tokens are billed but not stored.

**Use/Avoid:** Avoid inferring chars-per-token from `usage` metadata in Claude Code transcripts. If a real token-based metric is ever needed, run an actual tokenizer (e.g. `tiktoken` or Anthropic's own token-counting) directly over the extracted visible text content instead.

## `stat -c '%Y %n'` prints mtime, not size

**Context:** Investigating Claude Code session file sizes to check whether opaque `signature` blobs (thinking-block crypto payloads) were dominating transcript size.

**Observation:** Ran `stat -c '%Y %n'` intending to check file sizes, misread the mtime-epoch-seconds column as a byte count (`1783216381` read as ~1.78GB). This produced a real wrong claim mid-session — "some sessions are ~1.7GB, dominated by signature bloat" — that had to be explicitly retracted after re-checking with `stat -c '%s %n'` (actual sizes were unremarkable, hundreds of KB to a few MB; `signature` overhead is real but only ~17%, not ~99%).

**Impact:** Easy mistake to repeat when skimming `stat` output quickly under multi-column format strings — the epoch-seconds mtime value and a plausible-looking large byte count look similar at a glance.

**Use/Avoid:** Use `stat -c '%s %n'` for size, not `%Y` (mtime). When a `stat`-derived number looks surprising (e.g. multi-GB session files), sanity-check with `wc -c` or `ls -la` before asserting it as fact.

## `find-handoff-candidates` gap thresholds are thin-sample judgment calls, not statistically validated

**Context:** `skills/find-handoff-candidates/_rtx/_handoff_scan.py` — `DEFAULT_GAP_THRESHOLD = {"claude": 100_000, "codex": 1_000_000}`.

**Observation:** Claude Code's threshold is based on ~300 real sessions over 5 days (reasonably sampled). Codex's threshold is based on only ~15-17 real sessions over the same window, with a bimodal distribution (a cluster near 300K-1M, then several outliers at 5M+) — much less trustworthy than the Claude read. Neither threshold was validated against actual "did this session need a handoff" ground truth, because no such labeled data exists; both were picked as "somewhere between median and p75" by eye.

**Impact:** Expect both thresholds, especially Codex's, to misbehave (over- or under-flag) as usage patterns shift or as Codex's own transcript verbosity changes between CLI versions. `skills/find-handoff-candidates/_rtx/_handoff_calibrate.py` was added specifically to make re-deriving these numbers cheap when that happens — it reproduces the exact ad hoc methodology used for the original 2026-07 calibration (which had otherwise only existed in a throwaway `/tmp` script and this conversation).

**Use/Avoid:** If `wrap-up`'s handoff-candidate flagging starts feeling wrong (too noisy or missing obvious cases), run `_rtx/_handoff_calibrate.py --days N` before assuming the underlying `gap_net_chars` mechanism is broken — it may just be a threshold tuning problem with an existing, cheap fix.

## `blueprint.yaml`'s `depends_on` says *that* a skill depends on another, never *what it actually uses*

**Context:** `skills/list-manager` — before changing `lists.py`'s filtered-`read` return shape (flat list → pruned ancestor-preserving tree), needed to know which callers would break. `daily-plan`'s `blueprint.yaml` lists `list-manager` under `depends_on`, but that tells you nothing about *which* interfaces or behaviors it actually relies on.

**Observation:** Had to dispatch an Explore agent to grep `daily-plan/_rtx/_day_model.py` for actual `lists.py`/`read-beautify` invocations before it was safe to change the contract. Turned out `plan_runtime.py` never calls list-manager's filtered `read`/`cloud-read` at all — it builds its own flat entry lists independently and only calls `read-beautify` on an already-flat temp file it wrote itself. The `depends_on` declaration gave zero signal toward that; it would have been just as easy to wrongly assume `plan_runtime.py` used the filtered-read path and skip checking.

**Impact:** Any future change to a shared skill's interface contract needs an explicit call-site trace (grep/Explore agent) across every declared dependent — `depends_on` in `blueprint.yaml` is a coarse "these skills are related" signal, not a "here is what would break" signal. Trusting it alone risks either missing a real break or doing unnecessary defensive work for a caller that was never actually exposed.

**Use/Avoid:** Before changing a shared script's output shape or CLI contract, grep/dispatch-search every skill that declares a `depends_on` on it for actual invocations of the specific interface being changed, rather than assuming the dependency list itself tells you the blast radius.
