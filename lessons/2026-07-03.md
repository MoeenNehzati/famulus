# 2026-07-03

## list-manager: jsonschema/pyrsistent env issue

**Context:** `skills/list-manager/scripts/lists.py` — `validate_list()`, `HAS_JSONSCHEMA` flag.

**Observation:** `jsonschema 4.23` has an undeclared runtime dep on `pyrsistent` that isn't installed by default under Anaconda Python 3.13 on this machine. The import of `jsonschema` raises `ModuleNotFoundError: No module named 'pyrsistent'`, which is caught at module load, leaving `HAS_JSONSCHEMA = False`.

**Impact:** Before the 2026-07-03 fix, `validate_list` silently returned when `HAS_JSONSCHEMA` was False — all mutating writes (init/create-entry/update/delete) saved data without schema validation. After the fix, `validate_list` hard-fails with a clear error. Symptom when the dep is missing: *every* mutating operation exits nonzero with "jsonschema is required for mutating operations but is not installed", and all mutating tests fail.

**Fix:** `pip install pyrsistent` — restores `HAS_JSONSCHEMA = True` and unblocks all mutating ops.

## list-manager: create-entry now auto-fills state and created

**Context:** `cmd_create_entry` in `scripts/lists.py`.

**Observation:** As of 2026-07-03, `create-entry` defaults `state = "incomplete"` and `created = <today ISO>` for any entry missing those fields. Callers (including email-triage) no longer need to supply them.

**Impact:** Skills that construct entries and pass them to `create-entry` can omit `state` and `created`; the script fills them before validation runs. Existing callers that supply them explicitly are unaffected (the defaults only trigger when the key is absent).
