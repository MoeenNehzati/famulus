# Interface Metadata Refactor Plan

Status: design draft as of 2026-07-12.

## Goal

Make blueprint metadata rich enough to support:

- generated documentation that helps users choose the right skill;
- graph generation from live blueprint data instead of hardcoded skill lists;
- validation of repeated naming conventions across skills;
- portability and safety checks around files, network access, authentication, and
  system state.

The metadata should be meaningful beyond documentation. It should describe what
each interface does, what it touches, and what requirement it satisfies.

## Core Design Decisions

### Interface-first metadata

Most new metadata belongs under individual interfaces, not at the skill top
level.

Reason:

- different interfaces in the same skill can read, write, authenticate, or
  mutate different things;
- interface-level metadata can be validated against `behavior_sources`,
  `dependencies`, `uses_interfaces`, and invocation bindings;
- skill-level documentation can be inferred from the union of interface
  metadata.

The skill-level layer should stay small, but `role`, `kind`, and
`display_description` should be required once introduced:

```yaml
role: productivity
kind: workflow
display_description: "Builds a daily plan from schedule, weather, and list state."
```

Do not author top-level `systems`, `auth`, `inputs`, or `outputs` unless a later
use case proves that inferred summaries are insufficient.

### One description field

Do not add both `description` and `user_requirement`.

For each interface, `description` should mean:

> the user-facing requirement or outcome this interface satisfies.

It should not be an implementation note. It should be concise enough for docs
and graph tooltips. A character limit should be enforced, likely 160 or 180
characters.

The `SKILL.md` frontmatter description has a different purpose: it helps route
user prompts to the skill. It can mention trigger conditions and invocation
cues. Interface `description` should instead explain what the interface does for
the user or system.

Example:

```yaml
description: "Builds a daily plan from calendar events, weather, and list state."
```

Not:

```yaml
description: "Use when the user asks to plan the day."
```

### Controlled vocabulary

Fields such as `content`, `format`, `medium`, `system`, `access`, and `auth`
must have controlled allowed values wherever possible. Avoid free-form strings
for fields that will be used for search, validation, graph generation, docs
badges, or cross-skill comparison.

Reason:

- LLM-authored blueprints otherwise drift into near-duplicates like
  `Google Drive`, `google-drive`, `gdrive`, and `drive`;
- generated docs and graphs need stable filters;
- validators should catch misspellings and unapproved synonyms early;
- adding a new value should be a deliberate schema/template update.

Design rule:

- closed enums by default;
- arrays of closed enums for multi-valued facets;
- free-form prose only for explanation fields such as `description` and
  `reason`;
- if a field is expected to appear in filters or graphs, it should not be
  free-form.

When a new value is needed, update the schema and documentation first, then
update blueprints. Do not let individual skills introduce new spellings ad hoc.

### Semantic content, separate format

Use `content` for the semantic thing being read, written, sent, updated, or
requested.

Do not encode file format into `content`.

Good:

```yaml
content: report
format: html
path: "_build/*.html"
```

Avoid:

```yaml
content: html-report
path: "_build/*.html"
```

When `path` has a clear suffix, `format` may be inferred in docs, but keeping an
explicit `format` field is useful when the representation is not obvious from a
path, such as email messages, calendar events, API responses, stdout streams, or
remote records.

### Path roots

Plain relative paths are relative to the skill directory.

Example:

```yaml
path: "_build/*.html"
```

means:

```text
skills/<skill>/_build/*.html
```

Use explicit prefixes for other roots:

- `$repo/` for repository-root-relative paths;
- `$home/` for user-home paths;
- `$tmp/` for temporary directory patterns;
- absolute paths only when unavoidable and justified.

This matches the existing blueprint convention that interface roots are relative
to the directory containing `blueprint.yaml` unless a special prefix is used.

## Direct IO Model

Every interface must declare `direct_io`.

This applies to both namespaces:

- `interfaces.machine.*`
- `interfaces.llm.*`

`direct_io` is immediate-only. It describes what the interface itself touches,
consumes, emits, requests, or mutates. It must not duplicate IO from dependency
interfaces. Whole-skill and whole-workflow requirements are generated by
recursively following version-pinned `uses_interfaces` edges.

Required shape:

```yaml
direct_io:
  reads: []
  writes: []
  network: []
owns_filesystem: []
```

Empty lists mean "reviewed and no direct IO of this kind." Omitting
`direct_io`, or omitting one of the three lists, means the blueprint is
incomplete.

`owns_filesystem` is required beside `direct_io`. It declares filesystem path
ownership for local-file boundaries. If an interface owns a path, only that
interface may write matching local-filesystem `direct_io.writes` entries, and
only that interface plus `allowed_readers` may read matching
local-filesystem `direct_io.reads` entries. Two different interfaces must not
own overlapping filesystem paths.

`behavior_sources` is the mechanical behavior-root contract used for audit
hashing and runtime dependency tracking. `direct_io` is the semantic
documentation/search/graph contract.

### Behavior Sources

Interfaces have two different modes of reading files.

Some reads are the subject being transformed, checked, rendered, or summarized:
a user-provided `.bib` file, a PDF being converted, a TeX document being
reviewed, or a YAML list file being updated. These reads are operational inputs.
Their contents change from call to call and should not by themselves mean the
interface contract changed. They belong in `direct_io`, not in
`behavior_sources`.

Other reads are reference material that controls how the interface behaves:
JSON Schema files, prompt/instruction Markdown, templates, policy files, parser
tables, examples, or skill-owned config files. Changes to these files can change
the behavior of the interface and should trigger review or recertification.
These belong in `behavior_sources`.

LLM interfaces declare `behavior_sources` directly. Machine interfaces declare
`invocation.behavior_sources` because the files are read by the invoked runtime.
Python imports, `_rtx` entrypoints, and dispatcher targets are discovered
mechanically and should not be duplicated in behavior sources.

### Immediate vs Transitive IO

Authors only write direct IO.

Example: if `daily-plan.machine.orchestrate` calls
`cloud-files.machine.plans-read`, the daily-plan interface declares that call in
`uses_interfaces`; it does not copy the Google Drive read in its own
`direct_io`.

Generated docs should expose two views:

- direct IO: only what this interface itself touches;
- transitive IO: direct IO plus recursive IO from `uses_interfaces`.

This prevents stale duplication. If `cloud-files.machine.plans-read` changes
backend, callers update automatically through graph traversal.

## Proposed Interface Shape

```yaml
interfaces:
  machine:
    scripts-orchestrate:
      version: 1
      description: "Builds a daily plan from calendar events, weather, and list state."
      uses_interfaces:
        - interface: g-calendar.machine.scripts-gcal
          version: 1
        - interface: get-weather.machine.fetch
          version: 1
        - interface: cloud-files.machine.plans-read
          version: 1
        - interface: cloud-files.machine.plans-write
          version: 1

      direct_io:
        reads:
          - medium: prompt
            access: read
            read_role: subject
            content: plan-request
            format: text
            sensitivity: user-private

        writes:
          - medium: stdout
            access: write
            content: plan
            format: markdown
            sensitivity: user-private

        network: []

      owns_filesystem: []

    weather-fetch:
      version: 1
      description: "Fetches weather data for a location and date range."

      direct_io:
        reads: []
        writes:
          - medium: stdout
            access: write
            content: forecast
            format: json
            sensitivity: user-private

        network:
          - medium: network-request
            system: weather-service
            access: request
            content: forecast
            format: json
            auth:
              kind: none
              mode: none
            sensitivity: user-private

      owns_filesystem: []

  llm:
    default:
      version: 1
      description: "Audit a mathematical proof for soundness gaps."
      binding:
        kind: skill_file
        path: SKILL.md
      behavior_sources: []

      direct_io:
        reads:
          - medium: prompt
            access: read
            read_role: subject
            content: proof
            format: text
            sensitivity: user-private
        writes:
          - medium: prompt
            access: write
            content: audit-record
            format: markdown
            sensitivity: derived-private
        network: []
      owns_filesystem: []
```

`direct_io.reads` and `direct_io.writes` describe semantic resource movement.
Put an entry in `reads` when the interface directly consumes information from a
resource: a prompt, stdin, a local file, a remote file, an API response, a log,
or existing system state. Put an entry in `writes` when it directly emits or
mutates information: stdout, stderr, a local file, a remote file, an email,
calendar event, config value, or system state.

`direct_io.network` records every immediate network boundary. If an interface
reads a remote file through Google Drive, sends SMTP, calls a weather API, or
fetches arXiv, it gets a `direct_io.network` entry even when the same operation
also appears under `direct_io.reads` or `direct_io.writes`.

That means `network` is not a third data-flow direction. It is the network
boundary annotation for remote effects. The read/write direction still belongs
in `reads` or `writes`.

Examples:

- local YAML read: `direct_io.reads` only, with `medium: local-filesystem`;
- stdout JSON output: `direct_io.writes` only, with `medium: stdout`;
- Google Drive download: `direct_io.reads` with `medium: remote-filesystem`,
  plus `direct_io.network` with `medium: network-request`,
  `system: google-drive`, and `access: download`;
- SMTP send: `direct_io.writes` with `medium: network-request`,
  `system: email-smtp`, `access: send`, plus a matching `direct_io.network`
  boundary entry.

### Reference definitions from interface metadata

Every machine and LLM interface should require `direct_io`:

```json
{
  "direct_io": { "$ref": "#/definitions/directIo" },
  "owns_filesystem": { "$ref": "#/definitions/filesystemOwnershipList" }
}
```

`directIo` should require all three lists:

```json
{
  "type": "object",
  "required": ["reads", "writes", "network"],
  "additionalProperties": false,
  "properties": {
    "reads": {
      "type": "array",
      "items": { "$ref": "#/definitions/directIoEntry" }
    },
    "writes": {
      "type": "array",
      "items": { "$ref": "#/definitions/directIoEntry" }
    },
    "network": {
      "type": "array",
      "items": { "$ref": "#/definitions/directIoEntry" }
    }
  }
}
```

Interface `direct_io` entries should use those definitions:

```json
{
  "type": "object",
  "required": ["medium", "access", "content", "sensitivity"],
  "additionalProperties": false,
  "properties": {
    "medium": { "$ref": "#/definitions/metadataMedium" },
    "access": { "$ref": "#/definitions/metadataAccess" },
    "system": { "$ref": "#/definitions/metadataSystem" },
    "content": { "$ref": "#/definitions/metadataContent" },
    "format": { "$ref": "#/definitions/metadataFormat" },
    "auth": { "$ref": "#/definitions/metadataAuthRequirement" },
    "sensitivity": { "$ref": "#/definitions/metadataSensitivity" },
    "path": { "type": "string", "minLength": 1 },
    "reason": { "type": "string", "minLength": 1 }
  }
}
```

`reason` stays free-form because it explains why a declared resource is touched.
It should not be used as a grouping key.

Filesystem ownership entries should use exact or regex matching:

```json
{
  "type": "object",
  "required": ["match", "path", "allowed_readers"],
  "additionalProperties": false,
  "properties": {
    "match": { "enum": ["exact", "regex"] },
    "path": { "type": "string", "minLength": 1 },
    "allowed_readers": {
      "type": "array",
      "items": { "$ref": "#/definitions/canonicalInterfaceName" },
      "uniqueItems": true
    },
    "reason": { "type": "string", "minLength": 1 }
  }
}
```

The ownership validator then enforces: only the owning interface may write a
matching local-filesystem `direct_io` path, and only the owner plus
`allowed_readers` may read one.

### Update process for a new allowed value

When a skill needs a new `system`, `content`, `format`, `auth`, `medium`,
`access`, `role`, `kind`, or `sensitivity` value:

1. Add the value to the appropriate enum in
   `references/blueprint/schema.json`.
2. Add a short explanation and example to `references/blueprint/guide.md`.
3. Add the value to `references/blueprint/template.yaml` if it is common enough
   to show in the commented template.
4. Add or update validator tests that prove unknown values fail and the new
   value passes.
5. Update docs/search examples if the value is useful for user-facing filters.
6. Only then migrate affected `skills/*/blueprint.yaml` files.

This keeps vocabulary changes reviewable and prevents one-off synonyms from
spreading across skills.

### Validator expectations

Add validators or schema tests for:

- unknown enum values fail;
- casing is normalized to lower-case dash-separated names;
- every machine and LLM interface has `direct_io`;
- every machine and LLM interface has `owns_filesystem`;
- every `direct_io` block has `reads`, `writes`, and `network`;
- ownership regexes compile and `allowed_readers` names known canonical
  interfaces;
- owned local-filesystem paths are written only by their owner and read only by
  their owner or allowed readers;
- file-like `direct_io.reads` and `direct_io.writes` paths use approved roots;
- unprefixed paths are skill-root-relative;
- `$repo/`, `$home/`, and `$tmp/` prefixes are recognized;
- path-like values are not placed in `content`;
- suffix-like values such as `pdf`, `html`, `json`, and `yaml` are not placed in
  `content` when they belong in `format`;
- auth values are not invented per skill.

## Relationship To Existing Fields

Existing fields remain the mechanical contract:

- `dependencies`
- `uses_interfaces`
- `behavior_sources` / `invocation.behavior_sources`
- invocation `kind` and `entrypoint`

New `direct_io` metadata is semantic and documentation/search oriented, but
should be validated against the mechanical fields where possible.

Examples:

- a behavior-shaping local file should appear in `behavior_sources`, not only
  in `direct_io.reads`;
- a subject input such as `<file>` should appear in `direct_io.reads`, not in
  `behavior_sources`;
- a `direct_io.network` entry should usually correspond to a declared runtime
  dependency, auth requirement, or permission path that performs network work;
- an interface that declares `auth` should be discoverable in docs and setup
  checks.

## Inferred Skill-Level Views

The generated docs/search layer should infer skill-level summaries from
interface metadata:

- direct skill systems: union of interface `direct_io.*.system`;
- transitive skill systems: recursive union through `uses_interfaces`;
- skill auth requirements: union of interface `direct_io.*.auth`;
- skill contents read/written: union of interface `direct_io.*.content`;
- skill formats: union of interface `direct_io.*.format` and
  `direct_io.*.formats`;
- skill risk/effect badges: inferred from interface `direct_io.*.access`,
  `direct_io.*.medium`, `direct_io.*.system`, and
  `direct_io.*.sensitivity`;
- skill setup badges: inferred from `auth`, `systems`, and runtime dependencies;
- graph nodes and edges: generated from `systems`, `content`, `access`, and
  `uses_interfaces`.

This keeps authors from duplicating summaries at the skill level.

## Documentation And Graph Uses

The metadata should support:

- richer `docs/skills.md` entries such as "Reads", "Writes", "Needs auth", and
  "Changes";
- generated lists like "skills that read PDFs", "skills that require OAuth",
  "skills that update remote state", or "skills that touch systemd";
- graph generation for skill taxonomy, remote systems, network access,
  file/content flow, and auth requirements;
- cross-platform validators that flag Linux-only systems in skills marked
  cross-platform.

`skill-taxonomy` can be generated mostly from skill-level `role` and `kind`.
Workflow/data-flow graphs should use interface `direct_io` and
`uses_interfaces`.
Architecture graphs such as runtime/deployment maps may still need repo-level
authored metadata because they describe relationships outside individual skill
blueprints.

## Migration Plan

1. Update `references/blueprint/schema.json` with the new required skill-level
   fields and interface-level `direct_io` / `owns_filesystem` fields.
2. Update `references/blueprint/template.yaml` with commented examples and
   controlled vocabulary guidance.
3. Update `references/blueprint/guide.md` and
   `references/skill-guidelines.md`.
4. Add validators for controlled vocabulary, behavior-source path roots, and
   consistency between file-like `direct_io` entries and filesystem ownership.
5. Add version-impact validation. When a relevant interface contract changes,
   the owning interface version should bump; callers must keep
   `uses_interfaces[*].version` pinned to the target version. This should detect
   both stale pins and changed contracts without version bumps.
6. Add stronger filesystem ownership validation. Exact and regex ownership
   paths should compile, normalize relative roots, reject overlapping owners,
   and enforce that matching local-filesystem reads/writes respect owner and
   `allowed_readers` rules.
7. Add a same-skill shared-state ownership model. The current
   `owns_filesystem` contract is interface-owner oriented: one interface owns a
   path, only that interface writes it, and named readers may read it. Real
   skills such as `email-triage` have a shared state family where several
   same-skill interfaces legitimately write related files such as a watermark,
   status sentinel, or log. Future implementation should model that explicitly,
   either by adding `allowed_writers` to ownership entries or by introducing a
   separate skill-local state block with declared readers and writers. Until
   then, leave these shared paths unowned rather than falsely naming one
   interface as the sole writer.
8. Prune the `content` enum after several real migrations. Remove values that
   are too field-level, too format-like, or unused; add only coarse semantic
   values that support docs/search/graphs.
9. Extend blueprint sync/docs tooling so generated docs can read interface
   metadata.
10. Migrate a small slice first:
   - `g-calendar`;
   - `get-weather`;
   - `cloud-files`;
   - `daily-plan`;
   - `recurring-tasks`.
11. Use the migrated slice to generate one new docs view or graph, likely a
   remote systems/auth graph.
12. Migrate the remaining skills once the vocabulary and validators have proven
   stable.

During migration, do not mechanically copy an old skill-level dependency list
onto every interface. Add a `uses_interfaces` edge only to the interface that
directly calls, dispatches to, or routes through the target interface. If the
current code path is unclear, leave the edge out and add a migration note rather
than over-declaring it. Over-declared edges make generated transitive IO,
permission summaries, and graphs falsely broad.

## Resolved Compatibility Decisions

- `category` remains required during the migration because current generated
  contract blocks and validators still depend on it.
- `role` and `kind` are required additions, not optional migration hints.
- `role` and `kind` are primary skill-level labels. Interface-level `kind` and
  secondary skill-level labels can be added later when the interface metadata
  block is introduced.
- `direct_io` is required on every machine and LLM interface.
- `direct_io` is immediate-only; transitive IO is generated from
  `uses_interfaces`.
- `direct_io.network` should record every network boundary, including
  network-backed remote reads and writes that also appear under
  `direct_io.reads` or `direct_io.writes`.
- `owns_filesystem` is required on every machine and LLM interface. Empty
  ownership means "this interface is not the authority for any filesystem
  path"; non-empty ownership grants exclusive write authority and an explicit
  read allowlist for matching local-filesystem `direct_io` paths.

## Pilot Learnings

### `email-client`

The metadata model worked for a broad user-facing client with many operations,
but only after `content` was kept coarse. Values such as `email`, `mailbox`,
`email-attachment`, `credential`, and `response` were useful. Field-level values
such as subject/body/title/date would have made graphs noisy and inconsistent,
so the validator now rejects common field suffixes.

Auth needed to be structured as `{kind, mode}`. A flat `auth:
gmail-oauth`-style string could not distinguish reading with existing
credentials from creating, updating, deleting, or checking credentials.

`owns_filesystem` was mostly empty for this skill. That is correct: sending or
reading email is not filesystem ownership. The lesson is that a required empty
ownership list is useful as an explicit "reviewed; no ownership" marker.

### `cloud-files`

This pilot separated local filesystem ownership from remote storage authority.
The Google Drive list/plan operations are best represented as:

- `direct_io.reads` / `direct_io.writes` with `medium: remote-filesystem`;
- `direct_io.network` with `system: google-drive`;
- access control through `allowed_callers` and path patterns.

They should not use `owns_filesystem`, because `lists-write` and `lists-delete`
can both mutate the same remote path family. Treating that as single-writer
local ownership would be false. `owns_filesystem` did make sense for
`write-config`, which is the declared writer for
`$home/.config/cloud-files/config.json`, with `ensure-oauth` as an allowed
reader.

The search tool exposed a docs problem: projections flattened interface
descriptions, systems, and contents. Updating `explain` to return every
matching predicate value made "how did this skill match?" useful, but generated
docs and graphs still need grouped interface-level projections.

### `g-calendar`

This pilot showed that API clients are not remote files. Calendar operations
should use `medium: network-request` with `system: google-calendar`, not
`remote-filesystem`. The same interface can legitimately have both reads and
writes because `scripts-gcal` supports agenda/search/get and
create/update/delete.

Filesystem ownership did fit the local credential file:
`setup-oauth` is the declared writer for
`$home/.config/g-calendar/credentials.json`, while `scripts-gcal` and
`ensure-oauth` are allowed readers. The user-provided
`$home/.config/g-calendar/client.json` is read but not owned by the skill,
because the user is the source of truth for that file.

This also suggests a possible future validator: if an interface reads a local
credential path that another interface owns, it should either be the owner or
appear in `allowed_readers`; that rule already works. If it reads a secret path
with no owner, that may be acceptable when the user owns the file, but it may
deserve a warning-level review later.

### `list-manager`

This pilot made the immediate/transitive distinction essential. Local list
interfaces directly read and write user-selected `<file>` paths, but they do not
own those paths. A list file argument is runtime user data, not a skill-owned
state file.

Cloud list interfaces should not duplicate Google Drive IO in their own
`direct_io`; they declare `uses_interfaces` pointing to
`cloud-files.machine.lists-read` and `cloud-files.machine.lists-write`.
Generated docs should derive the transitive Google Drive effects from those
edges. This keeps `list-manager` honest: its immediate IO is patch/stdin/stdout
and local validation/rendering, while remote storage belongs to `cloud-files`.

This pilot also showed that `uses_interfaces` should be the dependency graph,
not a second layer beneath top-level skill dependencies. Each edge should carry
the target interface and required version.

### Cross-Pilot Conclusions

- `owns_filesystem` should remain local-filesystem only. Remote services need
  their own authority model based on interface access control, path patterns,
  and dependency edges.
- Required empty lists are useful. They force authors to decide that an
  interface has no immediate IO or no filesystem ownership.
- `direct_io.network` is not redundant with reads/writes. It is the graph edge
  for external service contact; reads/writes describe the semantic resource
  moved or mutated.
- Format families should stay finite. Use `format` for one format and
  `formats` for a finite list, with `path_match: glob` and final suffix families
  such as `*.{md,pdf}` when the path itself describes the family. This avoids
  broad `mixed` values while keeping docs/search/graphs mechanically useful.
- `uses_interfaces` is the right mechanism for transitive IO. Without it,
  callers either omit important effects or duplicate callee metadata.
- The search API needs a grouped projection mode, likely "select each matching
  interface with these child fields", for docs and graphs. Flat wildcard
  projection is acceptable for simple filtering but not for readable generated
  documentation.
- Validator coverage should next target local filesystem IO path/root
  consistency against filesystem ownership, plus recursive version-impact
  checks over `uses_interfaces`.
