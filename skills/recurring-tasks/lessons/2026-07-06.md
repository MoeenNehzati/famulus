# Lessons from Recurring-Tasks Refactoring

## Path-Clobbering Bug: How Test Installations Break Production

**Context:** Systemd recurring jobs were failing with `invoke-agent.sh: No such file or directory` pointing to temporary paths like `/tmp/codex-install-*/install-home/.claude/skills/...`

**Root Cause:** Test installations (like Codex installations in temporary directories) call `install-assistant-tools` with `--home /tmp/temp-dir/`. This:
1. Generates an `env.sh` with paths baked in
2. Sets `AI_AGENT_COMMAND_TEMPLATE` in the *live* systemd user environment (not scoped to the temp home)
3. After the test cleans up, the environment variable points to a nonexistent path
4. All recurring jobs break silently until manually fixed

The guard in `install_ai_agent_env()` checks `is_real_home` and skips setting the live environment if `--home != real $HOME`, but the implementation was not being followed.

**Solution:** Never store absolute paths in environment variables that persist across test installations. Instead:
- Put `invoke-skill` (the launcher) on PATH via `install-assistant-tools`
- Set `AI_AGENT_COMMAND_TEMPLATE = "invoke-skill {skill}"` (just a command name)
- The launcher resolves `$CLAUDE_HOME` at runtime
- Test installations generate their own `invoke-skill` in their temp bin dir and don't affect the real one

**Impact:** Prevents silent job failures from test pollution. Makes the system resilient to temporary installations that come and go.

**Implementation Details:**
- `sync-units.py` explicitly sets PATH in systemd ExecStart: `export PATH=... && <command>`
- `invoke-skill` (in managed bin dir) calls `invoke-agent.sh` with `$CLAUDE_HOME` resolved at runtime
- Systemd doesn't load shell rc files, so PATH must be set explicitly in the command, not relied on from bash login shells

## Architecture Simplification: Why Consolidation Matters

**Observation:** The original code had separate scripts for `enable-job.py`, `disable-job.py`, `test-job.py`, `view-logs.sh`, `status.sh`, plus `invoke-agent.sh`, `run-skill.sh`, and per-job runner scripts.

**Issue:** Each script did one thing, which seemed modular but created:
- 6-layer invocation chain: runner → run-skill → invoke-agent → assistant → skill
- Per-job runner script generation (duplication)
- Maintenance burden: fix a bug in one place, apply same fix to multiple scripts
- Harder to trace where things go wrong (which layer failed?)

**Solution:** Consolidate into `manage-job.py` with subcommands, simplify the invocation chain to 2 layers.

**Impact:** Easier to understand, test, and maintain. A new developer can read the entire flow in one script instead of tracing through 6 files.

**Use:** Next time you see multiple single-purpose scripts doing similar operations (enable, disable, run tests, view output), consider consolidating them with subcommands.

## Language Choice: Python Over Bash for Recurring Tasks

**Observation:** Original `healthcheck.sh` was 279 lines of bash. Refactored `healthcheck.py` is 171 lines.

**Why the difference:** Python's `subprocess`, `Path`, `datetime`, and structured data handling make the logic clearer. Bash requires escaping, quoting workarounds, and indirect state management.

**Impact:** The Python version is easier to read, extend, and test. It's cross-platform without bash-isms like `readlink -f` (doesn't work on macOS the same way).

**Best bet:** For anything that needs to run on different systems or has more than 50 lines of shell logic, write it in Python. Systemd units and short setup scripts can stay in bash.

## Why Explicit PATH in Systemd Units

**Context:** Systemd ExecStart does not source ~/.bashrc or ~/.zshrc. It starts bash in a minimal environment.

**Wrong approach:** Relying on `bash -lc` to load PATH from shell startup files means:
- PATH depends on shell configuration existing at runtime
- Moving to a different system changes behavior
- Hard to debug: "works on my machine" but not in CI or different shells

**Right approach:** Export PATH directly in the ExecStart command:
```
ExecStart=/bin/bash -c 'export PATH=... && <command>'
```

**Impact:** Units are portable, don't depend on shell rc files, and behavior is explicit in the unit file.

## What Not to Do: Test Infrastructure with Temp Directories

**Failed path:** The original `test-live-job.py` created temporary skill directories and temporary installations to test end-to-end behavior.

**Problem:** 
- Temporary installations call `install-assistant-tools --home /tmp/...`
- This sets environment variables in the *live* systemd session
- After cleanup, the live system is broken
- Tests should not mutate production state

**Solution:** Use unit tests of sync-units logic instead. Test YAML parsing, unit file generation, and cron-to-OnCalendar conversion without creating temp directories or installations.

**Impact:** Tests are fast, isolated, and don't risk breaking the real system.
